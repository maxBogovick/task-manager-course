# Урок 1. Типичные Ошибки И Ловушки

## Зачем нужен этот файл

Первый урок часто кажется простым.

Из-за этого у студента быстро возникает ложное ощущение:

“Я все понял”.

Но на практике именно в первом endpoint-е закладываются хорошие или плохие привычки.

Этот файл нужен, чтобы сразу отловить типичные ошибки мышления и кода.

---

## Ошибка 1. Пытаться понять Spring как набор аннотаций

Неправильная мысль:

“Надо просто запомнить, где ставится `@RestController`, `@Service` и `@GetMapping`.”

Почему это ошибка:

Если студент запоминает только синтаксис, он не понимает:

- какую проблему решает аннотация
- кто ее обрабатывает
- что было бы без нее
- как меняется поведение приложения

Правильная мысль:

Аннотация должна объясняться не как наклейка, а как подключение конкретного механизма.

---

## Ошибка 2. Класть всю логику в controller

Типичный код:

```java
@RestController
public class StatusController {

    @GetMapping("/api/orchestrator/status")
    public Map<String, Object> status() {
        Map<String, Object> result = new HashMap<>();
        result.put("runningTasks", 0);
        result.put("status", "ACTIVE");
        return result;
    }
}
```

Почему это ошибка:

- controller берет на себя лишнюю ответственность
- нет service-слоя
- код быстро начинает разрастаться
- студент привыкает, что HTTP-слой может делать все подряд

Как правильно:

- controller принимает запрос
- service делает use case
- DTO описывает ответ

---

## Ошибка 3. Возвращать `Map<String, Object>` вместо DTO

Почему новичкам нравится `Map`:

- быстро
- гибко
- не нужно создавать отдельный класс

Почему это плохая привычка:

- нет явного контракта
- легко допустить опечатку
- сложнее читать код
- слабее типобезопасность

Как правильно:

Использовать отдельный response DTO.

В первом уроке:

```java
public record OrchestratorStatusResponse(int runningTasks, String status) {}
```

---

## Ошибка 4. Создавать зависимости через `new`

Типичный код:

```java
private final TaskRunService taskRunService = new TaskRunService();
```

Или:

```java
private final TaskRunService taskRunService = new TaskRunService(new TaskOrchestrator());
```

Почему это ошибка:

- controller сам занимается wiring
- растет связанность
- сложнее тестировать
- архитектура перестает быть управляемой контейнером

Как правильно:

Зависимость должна приходить через constructor injection.

---

## Ошибка 5. Путать Lombok и DI

Очень частое заблуждение:

“`@RequiredArgsConstructor` внедряет зависимости.”

Это неверно.

Что делает Lombok:

- генерирует конструктор

Что делает Spring:

- создает объекты
- вызывает конструктор
- передает зависимости

Если студент не различает эти роли, он пока не понимает механику проекта.

---

## Ошибка 6. Путать Spring, Spring Boot и Spring Web

Частая проблема:

студент использует эти слова как синонимы.

Правильное различие:

- `Spring Framework` — большой фундамент
- `Spring Boot` — удобный запуск и автоконфигурация
- `Spring Web` — HTTP-слой

Если это не разделять, дальше начнется путаница почти во всех темах.

---

## Ошибка 7. Думать, что `record` — просто короткий класс

Это неполное понимание.

Почему `record` полезен:

- он выражает идею неизменяемых данных
- хорошо подходит для DTO
- убирает лишний бойлерплейт

Ошибка в мышлении:

видеть в `record` только сокращение синтаксиса и не видеть его смысл как модели данных.

---

## Ошибка 8. Не видеть разницу между “работает” и “хорошо спроектировано”

Очень опасная ошибка для старта.

Да, код может работать и в таком виде:

- все в controller
- `Map<String, Object>`
- `new` прямо в классе
- строковый JSON руками

Но инженерный вопрос звучит не так:

“Можно ли заставить это работать?”

А так:

“Можно ли это читать, тестировать, расширять и безопасно менять?”

---

## Ошибка 9. Не понимать, что именно Spring делает автоматически

Если студент видит только конечный код, у него возникает ощущение магии.

Он должен уметь перечислить, что именно Spring делает в первом уроке:

1. находит beans
2. создает beans
3. связывает зависимости
4. находит controller-методы
5. связывает URL с методами
6. вызывает метод по HTTP-запросу
7. превращает DTO в JSON
8. возвращает HTTP-response

Если этого списка нет в голове, понимание еще поверхностное.

---

## Ошибка 10. Слишком рано усложнять пример

Новичок может захотеть сразу добавить:

- базу данных
- scheduler
- полноценный orchestrator
- status enum
- logging everywhere
- обработку ошибок

Для урока 1 это ошибка.

Первая цель урока узкая:

понять базовый flow:

```text
HTTP -> controller -> service -> DTO -> JSON
```

Все остальное позже.

---

## Ошибка 11. Игнорировать чистую Java версию

Некоторые студенты думают:

“Зачем смотреть вариант без фреймворка, если мы все равно используем Spring?”

Это ошибка.

Pure Java версия нужна, чтобы понять:

- какую инфраструктуру заменяет Spring
- почему фреймворк вообще полезен
- что именно скрыто за аннотациями

Без этого Spring воспринимается как магический черный ящик.

---

## Ошибка 12. Не уметь объяснить код словами

Студент часто узнает код глазами, но не может его объяснить.

Признак неполного понимания:

человек видит `@RestController` и `@GetMapping`, но не может словами объяснить, что произойдет после HTTP-запроса.

Правильный уровень:

уметь устно восстановить весь flow запроса.
