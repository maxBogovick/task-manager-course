# Практика К Уроку 1

## Цель практики

На практике ты должен своими руками прийти к тому же результату, который разбирался в основном материале урока.

Финальная цель:

реализовать endpoint

`GET /api/orchestrator/status`

который возвращает JSON:

```json
{
  "runningTasks": 0,
  "status": "ACTIVE"
}
```

---

## Что ты должен отработать руками

В этой практике ты не просто повторяешь код.

Ты должен закрепить:

1. запуск Spring Boot приложения
2. создание controller
3. создание service
4. использование `record` для response DTO
5. constructor injection через Lombok
6. разделение ответственности между слоями

---

## Конечный практический результат

К концу практики у тебя должно быть:

1. приложение, которое стартует
2. controller с рабочим `GET` endpoint
3. service, в который вынесена логика
4. DTO на `record`
5. JSON-ответ в браузере или через `curl`

---

## Шаг 1. Найди целевой flow

Сначала зафиксируй в голове полный путь запроса:

```text
HTTP GET /api/orchestrator/status
        ->
controller
        ->
service
        ->
response DTO
        ->
JSON
```

Твоя задача на практике: не потерять ни одно звено этой цепочки.

---

## Шаг 2. Найди и прочитай целевые классы

Открой и прочитай:

1. [TaskManagerApplication.java](../../src/main/java/com/taskmanager/TaskManagerApplication.java)
2. [TaskExecutionController.java](../../src/main/java/com/taskmanager/controller/TaskExecutionController.java)
3. [TaskRunService.java](../../src/main/java/com/taskmanager/service/TaskRunService.java)
4. [OrchestratorStatusResponse.java](../../src/main/java/com/taskmanager/dto/OrchestratorStatusResponse.java)

После чтения попробуй без подсказки ответить:

- какой класс принимает HTTP-запрос
- какой класс хранит прикладную логику
- какой класс описывает ответ

---

## Шаг 3. Выпиши роли классов

Напиши для себя три короткие формулировки:

### Controller

Принимает HTTP-запрос и делегирует работу.

### Service

Выполняет прикладную логику use case.

### DTO

Описывает данные, которые уходят клиенту.

Если ты не можешь своими словами проговорить эти три роли, не переходи дальше.

---

## Шаг 4. Восстанови endpoint по памяти

Не копируй сразу код.

Сначала попробуй по памяти написать:

- controller
- service
- DTO

Потом сравни со структурой проекта.

Цель этого шага:

не механически перепечатать, а понять структуру.

---

## Шаг 5. Напиши controller

Требования:

1. класс должен быть REST-controller
2. должен иметь общий префикс `/api`
3. должен содержать `GET` endpoint `/orchestrator/status`
4. не должен сам вычислять ответ
5. должен делегировать работу service-слою

Проверь себя:

- есть ли `@RestController`
- есть ли `@RequestMapping("/api")`
- есть ли `@GetMapping("/orchestrator/status")`
- нет ли в controller лишней логики

---

## Шаг 6. Напиши service

Требования:

1. класс должен быть Spring service-bean
2. должен возвращать `OrchestratorStatusResponse`
3. должен быть отдельным от controller

Важно:

на этом шаге не нужно усложнять код.
Нам важна архитектурная форма:

```text
controller -> service
```

---

## Шаг 7. Напиши response DTO через `record`

Требования:

1. DTO должен называться `OrchestratorStatusResponse`
2. должен содержать:
   - `runningTasks`
   - `status`
3. должен быть реализован через Java `record`

После этого ответь себе:

- почему здесь `record` лучше обычного класса
- почему здесь не нужен `Map<String, Object>`

---

## Шаг 8. Добавь Lombok правильно

В controller и service используй:

- `@RequiredArgsConstructor`
- при необходимости `@Slf4j`

Но обязательно проговори себе:

- Lombok не внедряет зависимости
- Lombok только генерирует конструктор или логгер
- зависимости внедряет Spring

---

## Шаг 9. Запусти приложение

Запусти проект и проверь, что приложение стартует без ошибок.

После старта вызови endpoint:

```bash
curl http://localhost:8080/api/orchestrator/status
```

Ожидаемая идея ответа:

```json
{
  "runningTasks": 0,
  "status": "ACTIVE"
}
```

---

## Шаг 10. Объясни код вслух

Это обязательная часть практики.

Ты должен уметь устно объяснить:

1. почему controller не должен считать статус сам
2. почему service вынесен отдельно
3. почему response оформлен отдельным DTO
4. что такое bean
5. что такое DI
6. зачем нужен `record`
7. что делает Lombok

Если ты не можешь объяснить это без чтения текста, значит практика еще не завершена.

---

## Обязательное упражнение: антипример и исправление

Сделай два варианта endpoint-а.

### Вариант 1. Новичковый

Сделай плохо:

- верни `Map<String, Object>`
- помести все в controller
- не делай отдельный DTO

### Вариант 2. Профессиональный

Переделай:

- вынеси логику в service
- замени `Map` на `record`
- используй DI

После этого письменно ответь:

1. Чем первый вариант плох?
2. Почему второй вариант легче поддерживать?
3. Что произойдет, если таких endpoint-ов станет 50?

## Вопросы для самопроверки

1. Почему endpoint начинается не с service, а с controller?
2. Почему `record` здесь лучше, чем обычный mutable-класс?
3. Что потеряется, если убрать service и оставить только controller?
4. Что именно делает Spring, когда видит `@RestController`?
5. Что именно делает Spring, когда видит `@Service`?
6. Почему `new TaskRunService()` внутри controller обычно плохая идея?
7. Что конкретно упростил Lombok в этом примере?
