# Урок 1. Шаблоны Программирования, Принципы Проектирования И Лучшие Практики

## Зачем нужен этот файл

Этот материал нужен, чтобы студент с самого первого урока видел не только “как написать working code”, но и:

- какой здесь используется шаблон программирования
- какой принцип проектирования он поддерживает
- какую проблему решает профессиональная структура
- как выглядела бы наивная реализация
- почему наивный вариант хуже

Это особенно важно в учебном курсе.

Если студент видит только готовый правильный код, он может решить, что это “просто стиль автора”.

Нужно показывать инженерную логику выбора:

1. сначала наивная реализация
2. потом ее минусы
3. потом улучшенная версия
4. потом связь с паттерном, принципом и best practice

---

## Главный методический принцип для всех следующих уроков

Когда в уроке появляется паттерн или принцип, его нельзя давать как абстрактную теорию.

Правильная последовательность:

1. показать простое наивное решение
2. дать студенту почувствовать проблему
3. перечислить минусы
4. переписать решение лучше
5. только после этого назвать паттерн или принцип

Иначе студент запомнит название, но не поймет, зачем оно вообще нужно.

---

## Первый пример урока 1

Наша задача та же:

`GET /api/orchestrator/status`

Ответ:

```json
{
  "runningTasks": 0,
  "status": "ACTIVE"
}
```

Теперь разберем ее как инженерный выбор между наивным и профессиональным решением.

---

## Шаг 1. Наивная реализация

Вот типичный наивный вариант:

```java
@RestController
public class StatusController {

    @GetMapping("/api/orchestrator/status")
    public Map<String, Object> getStatus() {
        Map<String, Object> result = new HashMap<>();
        result.put("runningTasks", 0);
        result.put("status", "ACTIVE");
        return result;
    }
}
```

Этот код может работать.

Это важно проговорить.

Ошибка новичка не в том, что он написал “неработающий код”.
Ошибка в том, что он выбрал слабую архитектурную форму.

---

## Шаг 2. Минусы наивной реализации

### Минус 1. Нет разделения ответственности

Controller:

- принимает HTTP
- сам решает use case
- сам собирает структуру ответа

Одна точка начинает отвечать сразу за несколько ролей.

### Минус 2. Нет явного контракта ответа

`Map<String, Object>` не выражает модель ответа явно.

Минусы:

- слабая читаемость
- легко ошибиться в ключе
- сложно рефакторить
- нет типобезопасности

### Минус 3. Плохая база для роста

Пока endpoint один, проблема почти незаметна.
Когда endpoint-ов станет много, controller начнет раздуваться.

### Минус 4. Студент не учится профессиональной структуре

Наивный код дает ложный сигнал:

“Если работает, значит так и надо”.

Но задача курса не просто научить писать working code.
Задача курса — научить строить поддерживаемый backend.

---

## Шаг 3. Переписываем профессионально

```java
@RestController
@RequestMapping("/api")
@RequiredArgsConstructor
public class TaskExecutionController {

    private final TaskRunService taskRunService;

    @GetMapping("/orchestrator/status")
    public ResponseEntity<OrchestratorStatusResponse> getOrchestratorStatus() {
        return ResponseEntity.ok(taskRunService.getOrchestratorStatus());
    }
}
```

```java
@Service
public class TaskRunService {

    public OrchestratorStatusResponse getOrchestratorStatus() {
        return new OrchestratorStatusResponse(0, "ACTIVE");
    }
}
```

```java
public record OrchestratorStatusResponse(int runningTasks, String status) {}
```

---

## Что именно улучшилось после переписывания

Сначала нужно назвать не термин, а практический эффект:

1. один класс перестал делать все сразу
2. роли кода стали разделены
3. контракт ответа стал явным
4. зависимость перестала создаваться внутри controller

Только после этого имеет смысл вводить инженерные названия.

## Как называется разделение ролей по слоям

### `Layered Architecture`

Это не “GoF-паттерн”, а архитектурный стиль организации приложения по слоям.

В нашем примере:

- `controller` — web-слой
- `service` — application-слой
- `DTO` — слой передачи данных наружу

### Зачем он нужен

Чтобы разные роли системы не смешивались в одном месте.

### Что было в наивной реализации

Слои были фактически склеены в одном controller.

### Что изменилось после переписывания

Появились границы ответственности.

---

## Как назвать идеи, которые мы только что увидели в коде

### 1. Когда один класс перестает делать слишком много

Простыми словами:

у класса должна быть одна главная роль.

Только после этого можно назвать термин:

`SRP — Single Responsibility Principle`

#### Как это выглядело в наивном варианте

Controller одновременно:

- принимает HTTP
- хранит прикладную логику
- формирует контракт ответа

#### Что изменилось после переписывания

- controller отвечает за HTTP-вход
- service отвечает за use case
- DTO отвечает за форму данных

### 2. Когда разные заботы не лежат в одной куче

Простыми словами:

разные заботы системы надо разделять.

Только после этого можно назвать термин:

`Separation of Concerns`

#### Где проблема была в наивной версии

HTTP-забота и прикладная забота смешаны.

#### Что изменилось в улучшенной версии

Каждая часть делает свою работу.

### 3. Dependency Inversion на базовом уровне

В первом уроке это пока не в полной академической форме, но студент уже должен увидеть важную мысль:

класс не должен сам создавать свои зависимости.

Когда controller получает service извне, а не создает его через `new`, код становится слабее связан.

### 4. Когда контракт API выражен явно

Это не формальный SOLID-принцип, но важная инженерная практика:

API должен иметь явный контракт.

`record OrchestratorStatusResponse(...)` выражает контракт явно.

`Map<String, Object>` — нет.

## Какие рабочие практики здесь получились

Сначала зафиксируем смысл:

- controller лучше держать тонким
- зависимости лучше передавать через конструктор
- DTO лучше, чем `Map`
- `record` хорошо подходит для простых моделей ответа
- HTTP-ответ полезно делать явным

Только после этого можно сказать, что это уже best practices.

### Best Practice 1. Thin Controller

Controller должен быть тонким.

Это значит:

- минимум логики
- максимум делегирования

Почему:

HTTP-слой должен оставаться простым и читаемым.

### Best Practice 2. Constructor Injection

Зависимости лучше передавать через конструктор.

Почему:

- обязательные зависимости видны сразу
- класс легче тестировать
- объект нельзя создать в полубитом состоянии

### Best Practice 3. DTO вместо `Map`

DTO делает API-контракт:

- читаемым
- типизированным
- устойчивым к изменениям

### Best Practice 4. Использование `record` для простых response-моделей

Если объект является просто переносчиком данных, `record` делает код:

- короче
- яснее
- безопаснее

### Best Practice 5. Явное управление HTTP-ответом через `ResponseEntity`

Даже в простом первом уроке студент должен видеть, что backend работает не просто с Java-объектом, а с HTTP-ответом.

## Сравнение “до” и “после”

### Наивный вариант

```java
@RestController
public class StatusController {

    @GetMapping("/api/orchestrator/status")
    public Map<String, Object> getStatus() {
        Map<String, Object> result = new HashMap<>();
        result.put("runningTasks", 0);
        result.put("status", "ACTIVE");
        return result;
    }
}
```

### Улучшенный вариант

```java
@RestController
@RequestMapping("/api")
@RequiredArgsConstructor
public class TaskExecutionController {

    private final TaskRunService taskRunService;

    @GetMapping("/orchestrator/status")
    public ResponseEntity<OrchestratorStatusResponse> getOrchestratorStatus() {
        return ResponseEntity.ok(taskRunService.getOrchestratorStatus());
    }
}
```

```java
@Service
public class TaskRunService {

    public OrchestratorStatusResponse getOrchestratorStatus() {
        return new OrchestratorStatusResponse(0, "ACTIVE");
    }
}
```

```java
public record OrchestratorStatusResponse(int runningTasks, String status) {}
```

### Что улучшилось

1. Появился явный service-слой.
2. Появился явный контракт ответа.
3. Controller перестал быть перегруженным.
4. Код легче читать и объяснять.
5. Архитектура лучше масштабируется.

## Как правильно называть это студенту

Нельзя говорить:

“Мы делаем так, потому что так профессиональнее.”

Этого недостаточно.

Нужно говорить причинно:

- наивная версия смешивает роли
- это нарушает разделение ответственности
- поэтому мы выделяем service
- DTO делает контракт явным
- constructor injection уменьшает связанность
- в итоге код легче развивать

То есть сначала причина, потом техника, потом название принципа.

## Как это переносится на следующие уроки

Этот же шаблон подачи нужно повторять дальше.

Например:

### Когда появится repository

Сначала:

- наивный доступ к данным прямо из controller или service

Потом:

- выделение repository

И только потом:

- разговор про слои, responsibility boundaries и persistence abstraction

### Когда появится executor architecture

Сначала:

- наивный `if-else` по типу задачи

Потом:

- интерфейс `TaskExecutor`
- отдельные реализации
- registry

И только потом:

- разговор про Strategy, OCP и plugin architecture

### Когда появится orchestration

Сначала:

- наивная логика запуска прямо в controller или service

Потом:

- отдельный orchestrator

И только потом:

- разговор про coordinator/facade role и границы use case orchestration

## Мини-шаблон для всех следующих уроков

Когда вы будете делать следующие материалы, полезно держать одну и ту же структуру:

### 1. Наивная реализация

Показать простой вариант.

### 2. Почему он недостаточен

Дать конкретные минусы.

### 3. Улучшенная реализация

Показать переписанный вариант.

### 4. Какой паттерн здесь появился

Назвать и объяснить.

### 5. Какие принципы здесь работают

Показать, а не просто перечислить.

### 6. Какие лучшие практики отсюда надо вынести

Сформулировать коротко и прикладно.
