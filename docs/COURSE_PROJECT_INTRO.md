# Введение В Учебный Проект TaskManager

## Зачем тебе этот проект

В этом курсе ты будешь изучать Spring Boot не на искусственных маленьких примерах, а на полноценном backend-приложении.

Этот проект называется `TaskManager`.

Его задача — хранить, запускать, отслеживать, повторно запускать и отменять задачи разных типов.

Проще говоря, это система, которая умеет:

- хранить описание задачи
- запускать задачу вручную
- запускать задачу по расписанию
- хранить историю запусков
- показывать статус выполнения
- отменять выполнение
- повторять неудачные выполнения

Это хороший учебный проект, потому что в нем есть не одна изолированная тема, а почти весь путь реального backend-приложения:

- REST API
- controller-ы
- service-слой
- DTO
- JPA
- PostgreSQL
- Flyway
- обработка ошибок
- планировщик
- асинхронное выполнение
- расширяемая архитектура через executors
- integration tests

Главная цель курса:

не просто понять этот код, а научиться создавать приложение такого же уровня самостоятельно.

---

## Что это за система простыми словами

Представь, что у тебя есть сервер, который умеет выполнять полезные действия.

Например:

- выполнить shell-команду
- отправить HTTP-запрос
- отправить сообщение в Telegram
- разобрать файлы по папкам

Но сервер должен не просто “что-то сделать один раз”.

Он должен уметь:

- сохранить описание задачи
- знать, включена задача или выключена
- уметь запускать задачу по кнопке
- уметь запускать задачу по cron-расписанию
- знать, сколько раз задача уже запускалась
- понимать, успешно ли завершилось выполнение
- уметь остановить зависшее выполнение
- автоматически повторить неудачный запуск

Именно это и делает `TaskManager`.

## Что ты увидишь в готовом приложении

К концу курса ты должен понимать, как построить backend, который умеет:

1. принимать HTTP-запросы от клиента
2. создавать и изменять сущности в базе данных
3. запускать бизнес-операции вручную и по расписанию
4. отслеживать состояние выполнения
5. отделять HTTP-слой от прикладной логики
6. расширяться без переписывания центра системы

То есть ты будешь учиться не просто “писать controller”, а строить систему из нескольких слоев и понимать, зачем каждый слой нужен.

## Какие главные сущности есть в проекте

В проекте есть две центральные сущности.

## 1. TaskDefinition

Это описание задачи.

Здесь хранится:

- имя задачи
- описание
- тип задачи
- JSON-конфиг
- cron-расписание
- включена задача или нет
- настройки ретраев
- таймаут
- пользователь, которому задача принадлежит

Проще говоря:

`TaskDefinition` отвечает на вопрос:

**что именно нужно запускать и как это запускать.**

## 2. TaskExecution

Это один конкретный запуск задачи.

Здесь хранится:

- какая задача была запущена
- статус выполнения
- номер попытки
- когда запуск начался
- когда закончился
- какой был output
- какая возникла ошибка

Проще говоря:

`TaskExecution` отвечает на вопрос:

**что произошло при конкретном запуске задачи.**

---

## Какой пользовательский сценарий поддерживает система

Вот типичный жизненный сценарий:

1. Клиент создает определение задачи через API.
2. Сервер сохраняет его в базе данных.
3. Клиент запускает задачу вручную или ждет cron-запуска.
4. Система создает запись о выполнении.
5. Оркестратор запускает нужный executor.
6. Выполнение либо завершается успешно, либо падает, либо отменяется, либо истекает по таймауту.
7. История выполнения сохраняется.
8. Если задача упала и для нее разрешены ретраи, система может попробовать запустить ее снова.

Когда ты понимаешь этот сценарий целиком, тебе уже легче понимать, зачем в проекте столько слоев и классов.

## Какие возможности есть у API

Ниже перечислены ключевые запросы, которые поддерживает система.

На старте курса тебе не нужно знать реализацию каждого из них.
Сейчас твоя задача — увидеть общую картину.

## 1. Работа с определениями задач

### `POST /api/tasks`

Создает новую задачу.

Что делает:

- принимает данные новой задачи
- валидирует вход
- сохраняет `TaskDefinition` в базе
- возвращает созданный объект в виде response DTO

### `GET /api/tasks`

Возвращает список задач.

Что делает:

- читает задачи из базы
- поддерживает пагинацию
- может фильтровать по типу задачи

### `GET /api/tasks/{id}`

Возвращает одну задачу по id.

### `PUT /api/tasks/{id}`

Обновляет существующую задачу.

### `DELETE /api/tasks/{id}`

Удаляет задачу.

### `PATCH /api/tasks/{id}/enabled`

Включает или выключает задачу.

### `GET /api/tasks/user/{userId}`

Возвращает задачи конкретного пользователя.

### `GET /api/tasks/types`

Возвращает список доступных типов задач.

Это важный endpoint, потому что он показывает расширяемость системы.

---

## 2. Запуск и управление выполнением

### `POST /api/tasks/{id}/run`

Запускает задачу вручную.

Что делает:

- находит определение задачи
- проверяет, что задача включена
- создает `TaskExecution`
- передает выполнение в оркестратор

### `POST /api/executions/{id}/cancel`

Пытается отменить выполнение.

Что делает:

- находит running execution
- выставляет флаг отмены
- завершает выполнение со статусом `CANCELLED`, если это возможно

---

## 3. История выполнений

### `GET /api/executions`

Возвращает список выполнений.

Может фильтровать по статусу.

### `GET /api/executions/{id}`

Возвращает детали одного выполнения.

### `GET /api/tasks/{id}/executions`

Возвращает историю выполнений конкретной задачи.

---

## 4. Мониторинг

### `GET /api/orchestrator/status`

Возвращает состояние оркестратора.

Это один из самых простых endpoint-ов в проекте, поэтому с него удобно начинать обучение.

---

## Какие типы задач поддерживаются

Система умеет запускать разные типы задач.

Это реализовано через отдельные executor-ы.

## 1. `SHELL_COMMAND`

Запускает shell-команду в операционной системе.

Примеры:

- `echo hello`
- `curl https://example.com`
- запуск локального скрипта

## 2. `HTTP_REQUEST`

Отправляет HTTP-запрос.

Подходит, если задача должна:

- вызвать внешний API
- отправить webhook
- получить данные из другого сервиса

## 3. `TELEGRAM_MESSAGE`

Отправляет сообщение через Telegram Bot API.

## 4. `FILE_ORGANIZER`

Разбирает файлы по папкам по заданным правилам.

Это особенно полезно как пример задачи, у которой есть более прикладная локальная логика.

## Как устроено приложение по слоям

Чтобы не утонуть в коде, важно сразу видеть общую архитектуру.

Проект устроен по слоям.

## 1. Controller layer

Классы:

- `TaskDefinitionController`
- `TaskExecutionController`

Что делают:

- принимают HTTP-запросы
- читают входные данные
- вызывают service-слой
- возвращают HTTP-ответ

Controller не должен хранить тяжелую прикладную логику.

## 2. Service layer

Классы:

- `TaskDefinitionService`
- `TaskExecutionService`
- `TaskRunService`
- `ConfigValidator`
- `ReconciliationService`

Что делают:

- содержат прикладную логику
- координируют работу разных частей системы
- решают use case уровня приложения

## 3. Repository layer

Классы:

- `TaskDefinitionRepository`
- `TaskExecutionRepository`

Что делают:

- работают с базой данных
- дают удобный способ читать и сохранять данные

## 4. Entity layer

Классы:

- `TaskDefinition`
- `TaskExecution`
- `ExecutionStatus`

Что делают:

- описывают данные, которые живут в базе

## 5. DTO layer

Классы из `dto/`:

- request DTO
- response DTO
- paged response
- status response

Что делают:

- определяют форму данных, которые приходят в API и уходят из API

## 6. Orchestrator layer

Классы:

- `TaskOrchestrator`
- `TaskSchedulerService`
- `TaskSubmittedEvent`

Что делают:

- управляют жизненным циклом выполнения задач
- запускают их асинхронно
- следят за таймаутами
- повторяют неудачные задачи
- запускают задачи по расписанию

## 7. Executor layer

Классы:

- `TaskExecutor`
- `TaskExecutorRegistry`
- `...Executor` реализации

Что делают:

- реализуют конкретное поведение для каждого типа задач

## Какой путь проходит запрос в приложении

На курсе мы будем много раз проходить один и тот же важный путь:

```text
HTTP request
    ->
Controller
    ->
Service
    ->
Repository
    ->
Database
    ->
Service
    ->
Response DTO
    ->
HTTP response
```

Это базовый request flow любого Spring Boot CRUD-приложения.

Позже к нему добавятся более сложные сценарии:

- асинхронный запуск
- события
- планировщик
- retry flow
- cancellation flow

## Какие программные шаблоны ты изучишь на этом проекте

На этом проекте будут раскрываться не абстрактные паттерны “ради теории”, а реальные шаблоны проектирования, которые используются в коде.

## 1. Layered Architecture

Приложение разбито на слои:

- controller
- service
- repository
- entity
- dto

Ты поймешь:

- зачем слои разделяют
- почему нельзя складывать всю логику в один класс
- как слои взаимодействуют друг с другом

## 2. Dependency Injection

Ты увидишь, как объекты получают зависимости извне, а не создают их сами.

Это даст понимание:

- beans
- IoC-контейнера
- почему `new` внутри business-классов часто плохая идея

## 3. Repository Pattern

Ты увидишь, как доступ к данным выносится в отдельный слой.

## 4. Service Layer

Ты увидишь, почему controller не должен хранить прикладную логику внутри себя.

## 5. DTO Pattern

Ты поймешь, почему entity не надо отдавать напрямую наружу.

## 6. Mapper

Ты увидишь, зачем нужен отдельный mapping между entity и DTO.

## 7. Facade

`TaskRunService` выступает как удобная точка входа в use case запуска и отмены задач.

## 8. Strategy

Каждый `TaskExecutor` реализует свой способ выполнения задачи.

## 9. Registry

`TaskExecutorRegistry` хранит и выдает подходящий executor по типу задачи.

## 10. Event-driven interaction

В проекте есть отправка событий при подаче задачи на выполнение.

## 11. Lifecycle / State thinking

Ты увидишь, как проектируется система, у которой есть жизненный цикл состояний:

- `PENDING`
- `RUNNING`
- `COMPLETED`
- `FAILED`
- `TIMED_OUT`
- `CANCELLED`

## Какие принципы программирования раскрывает проект

Помимо конкретных паттернов, этот проект очень хорошо подходит для изучения базовых инженерных принципов.

## 1. Single Responsibility Principle

Каждый класс должен отвечать за одну основную область ответственности.

Примеры:

- controller — за HTTP
- service — за прикладную логику
- repository — за доступ к данным

## 2. Separation of Concerns

Нужно разделять:

- web-слой
- бизнес-логику
- инфраструктуру
- хранение данных

## 3. Open/Closed Principle

Система должна расширяться без переписывания центральной логики.

В этом проекте это хорошо видно на executor-архитектуре:

- можно добавить новый тип задачи
- не нужно переписывать все приложение заново

## 4. Programming To Abstractions

Работа через интерфейсы и контракты делает систему гибче.

Особенно это видно на `TaskExecutor`.

## 5. Thin Controller, Rich Service

Controller должен быть тонким.
Он принимает запрос и делегирует работу.

## 6. Fail Fast

Система старается валидировать вход как можно раньше.

## 7. Explicit Contracts

DTO, response-модели и валидация делают API более предсказуемым.

## 8. Controlled Side Effects

Запуск процессов, HTTP-вызовы, Telegram, файловые операции и фоновые задачи должны быть изолированы и управляемы.

## Как мы будем проходить проект

Курс будет построен постепенно.

Мы не будем сразу бросаться во все подсистемы.

Логика обучения будет такой:

1. Берем самый простой endpoint.
2. Полностью разбираем его путь через слои.
3. Понимаем controller, service, DTO и beans.
4. Потом добавляем repository и entity.
5. Потом собираем полный CRUD flow.
6. Только после этого переходим к сложным вещам:
   - execution lifecycle
   - orchestrator
   - scheduling
   - retries
   - executors

Это важно, потому что сначала нужно научиться уверенно понимать обычный request flow, а потом уже переходить к более сложной архитектуре.
