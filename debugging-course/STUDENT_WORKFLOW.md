# Рабочий Процесс Студента

Используй этот алгоритм в каждом lab.

## 1. Зафиксируй симптом

Запиши, что именно не работает:

- какой endpoint;
- какой статус ответа;
- какой статус execution;
- какое поле не совпадает;
- какой exception;
- какой тест падает.

## 2. Найди точку входа

Для HTTP-запроса начинай с controller:

- `TaskDefinitionController`;
- `TaskExecutionController`.

Для фоновой логики начинай с service:

- `TaskOrchestrator`;
- `TaskSchedulerService`;
- `TaskExecutionService`.

## 3. Пройди поток выполнения

Запиши цепочку вида:

```text
HTTP -> Controller -> Service -> Repository -> Entity
```

или:

```text
Scheduler -> Orchestrator -> Executor -> TaskExecutionService -> Repository
```

## 4. Сформулируй гипотезу

Хорошая гипотеза звучит так:

```text
Я думаю, что статус меняется неверно в TaskExecutionService.finishExecution,
потому что cancelExecution уже поставил CANCELLED, но фоновой поток может позже записать FAILED.
```

Плохая гипотеза:

```text
Наверное, Spring сломался.
```

## 5. Напиши регрессионный тест

Тест должен падать на баге и проходить после фикса.

Если баг слишком асинхронный, можно сначала сделать ручную проверку, но в отчете нужно объяснить, почему тест не был добавлен.

## 6. Сделай минимальный фикс

Минимальный фикс:

- меняет причину дефекта;
- не переписывает соседнюю архитектуру;
- не добавляет новую абстракцию без необходимости;
- сохраняет существующие tests green.

## 7. Проверь

Минимум:

```bash
mvn test -Dtest="TaskManagerApplicationTests#имяМетода"
```

Лучше:

```bash
mvn test
```

## 8. Напиши вывод

В выводе должны быть:

- причина бага;
- почему тест ловит баг;
- почему фикс корректен;
- какой риск остался.
