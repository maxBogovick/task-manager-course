# Правила Курса

Эти правила защищают фокус курса. Если упражнение не помогает читать, диагностировать и фиксить существующий код, оно не входит в этот курс.

## Основное правило

Каждый lab должен тренировать один главный навык:

- чтение потока выполнения;
- локализация дефекта;
- воспроизведение поведения тестом;
- минимальный фикс;
- объяснение результата.

## Правила студента

1. Нельзя начинать с переписывания кода.
2. Нельзя менять архитектуру, пока не доказан дефект.
3. Нельзя фиксить без понятного симптома.
4. Нельзя считать фикс готовым без теста или явной проверки.
5. Нельзя удалять существующее поведение, если оно не связано с багом.
6. Нельзя править больше модулей, чем требует причина дефекта.
7. Нельзя скрывать неизвестное: если причина не ясна, нужно записать гипотезу.

## Правила преподавателя

1. Не давать файл с багом сразу, если lab тренирует поиск.
2. Не принимать ответ "я поменял и заработало" без объяснения причин.
3. Не требовать идеального production-решения, если lab про локальный дефект.
4. Оценивать ход расследования, а не только финальный diff.
5. Требовать короткий письменный отчет после каждого lab.

## Формат ответа студента

В конце каждого lab студент должен сдать:

```text
Симптом:
Ожидаемое поведение:
Фактическое поведение:
Файлы, которые я прочитал:
Главная гипотеза:
Причина:
Тест:
Фикс:
Почему фикс минимальный:
Какой риск остался:
```

## Что нельзя добавлять в labs

- большие рефакторинги;
- новые фреймворки;
- "улучшения на будущее" без дефекта;
- задания на UI, если lab про backend;
- задания на оптимизацию без измеримого симптома;
- изменения Docker, Maven или окружения без учебной причины.
