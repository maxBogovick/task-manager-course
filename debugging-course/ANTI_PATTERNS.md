# Антипаттерны

Эти ошибки чаще всего ломают обучение debugging.

## Переписать вместо понять

Плохой признак:

```text
Я переписал service, потому что так проще.
```

Правильный вопрос:

```text
Какой конкретно дефект требует изменения service?
```

## Фикс без симптома

Плохой признак:

```text
Мне показалось, что так лучше.
```

Правильный вопрос:

```text
Какой тест или request показывает проблему?
```

## Тест после угадывания

Плохой признак:

```text
Сначала поменяю, потом подгоню тест.
```

Правильный вопрос:

```text
Как тест должен упасть на текущем поведении?
```

## Лечение в controller

Плохой признак:

```text
Я добавлю if в controller, чтобы status был правильный.
```

Правильный вопрос:

```text
Это presentation concern или lifecycle/business concern?
```

## Hardcoded plugin logic

Плохой признак:

```text
if taskType == SHELL_COMMAND в ConfigValidator.
```

Правильный вопрос:

```text
Как существующий executor plugin mechanism должен дать config class?
```

## Игнорирование async race

Плохой признак:

```text
Cancel уже вызвали, значит фоновый поток точно остановился.
```

Правильный вопрос:

```text
Что будет, если background thread запишет результат после cancel?
```

## Слишком широкий diff

Плохой признак:

```text
Я поправил еще форматирование и переименовал классы.
```

Правильный вопрос:

```text
Какие строки необходимы для исправления причины?
```
