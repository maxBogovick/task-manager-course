# Вопросы, Которые Стоило Задать

Ниже вопросы, которые полезно задать до создания курса, и лучшие ответы для этого проекта.

## 1. Студент должен строить проект с нуля или читать готовый код?

Лучший ответ: читать готовый код.

В репозитории уже есть staged training для строительства системы. Этот курс должен закрыть другой навык: вход в существующий production-like код.

## 2. Нужно ли специально ломать код?

Лучший ответ: да, но аккуратно.

Лучший формат - отдельные labs с описанием симптома и optional patch-файлами в будущем. В первой версии достаточно заданий, которые указывают, какой тип дефекта внести или расследовать. Реальный код в `src` не должен быть сломан заранее.

## 3. Нужно ли давать решения?

Лучший ответ: да, но решения должны объяснять ход мысли, а не только diff.

Студент должен видеть:

- как найти точку входа;
- как исключать неверные гипотезы;
- почему выбранный фикс минимален;
- какие тесты подтверждают поведение.

## 4. Нужно ли требовать тест перед фиксом?

Лучший ответ: да.

Исключение допустимо только для очень сложных асинхронных сценариев. Тогда студент должен описать ручную проверку и предложить, какой тест добавить позже.

## 5. Как не превратить курс в теорию Spring?

Лучший ответ: каждая тема Spring должна появляться только из реального бага.

Например, transactions объясняются через `TaskOrchestrator.submitTask()` и `@TransactionalEventListener`, а не отдельной лекцией на 30 страниц.

## 6. Нужно ли учить git?

Лучший ответ: немного, только в рамках debugging workflow.

Полезный минимум:

```bash
git status
git diff
git diff -- src/main/java/...
```

Но это не git course.

## 7. Какой главный критерий качества курса?

Лучший ответ: после курса студент может получить баг-репорт и самостоятельно дойти до причины.

Не "знает все классы", а умеет двигаться по системе.

## 8. Нужно ли давать студенту архитектурную карту?

Лучший ответ: да, но краткую.

Карта должна ускорять вход, а не заменять чтение кода. Поэтому она лежит отдельно в `architecture/`.

## 9. Нужно ли покрывать frontend?

Лучший ответ: нет в первой версии.

В `src/main/resources/static` есть UI, но цель этого трека - backend debugging. UI можно добавить отдельным курсом позже.

## 10. Что ты забыл предложить?

Лучшие дополнения:

- ввести "debugging journal" для каждого lab;
- добавить рубрику оценки;
- добавить режимы сложности;
- добавить anti-pattern list;
- добавить финальный production-style lab без подсказок;
- позже добавить `introduce-bug.patch` для каждого lab;
- позже добавить автопроверку структуры отчета студента.
