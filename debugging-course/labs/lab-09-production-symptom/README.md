# Lab 09 - Production Symptom

## Цель

Научиться расследовать баг по неполному production-style описанию.

## Симптом

Пользователь сообщает:

```text
Я запустил задачу, нажал cancel почти сразу, UI показал cancelled,
но через несколько секунд execution в истории стал failed.
```

## Файлы для чтения

Студент должен выбрать сам.

## Задание

1. Сформулируй 3 гипотезы.
2. Найди endpoint cancel.
3. Найди все места, где execution status может измениться после cancel.
4. Определи вероятную race condition.
5. Предложи regression test.
6. Предложи минимальный фикс.
7. Объясни, почему фикс не должен быть в controller.

## Ограничения

- преподаватель не дает список файлов первые 20 минут;
- нельзя начинать с фикса;
- нельзя менять UI;
- нельзя считать bug невозможным только потому, что тест сейчас проходит.

## Подсказки

### Уровень 1

Ищи все usages `ExecutionStatus.CANCELLED`.

### Уровень 2

Ищи все places, где вызывается `setStatus`.

### Уровень 3

Фоновый поток и HTTP cancel request могут конкурировать.

## Эталонное рассуждение

Это race между `cancelExecution` и `finishExecution`. Cancel request ставит `CANCELLED`, но executor thread может позже попытаться записать result. Защита должна быть в write method, который завершает execution, потому что именно там централизованно меняется final status. Controller не подходит: он не контролирует background write.

## Проверка

```bash
mvn test
```

Лучший regression test запускает долгую команду, отменяет execution и poll-ит финальный статус, ожидая, что он останется `CANCELLED`.
