# Лекция Stage 01

## Что мы строим на этом этапе

На этом этапе появляется первый настоящий доменный MVP:

- создание задачи;
- получение задачи по id;
- получение списка задач;
- сохранение данных в базе.

Это первый момент, когда студент видит полный вертикальный срез backend.

Кодовая база этапа:

- [TaskDefinition.java](./start/src/main/java/com/taskmanager/entity/TaskDefinition.java)
- [TaskDefinitionController.java](./start/src/main/java/com/taskmanager/controller/TaskDefinitionController.java)
- [TaskDefinitionService.java](./start/src/main/java/com/taskmanager/service/TaskDefinitionService.java)
- [TaskDefinitionRepository.java](./start/src/main/java/com/taskmanager/repository/TaskDefinitionRepository.java)
- [TaskDefinitionMapper.java](./start/src/main/java/com/taskmanager/mapper/TaskDefinitionMapper.java)
- [V1__create_task_definitions.sql](./start/src/main/resources/db/migration/V1__create_task_definitions.sql)

## Зачем нужен этот этап

После Stage 00 студент уже знает, как устроен минимальный backend-каркас.

Теперь нужно показать, как backend становится системой, которая:

- принимает данные извне;
- валидирует их;
- преобразует их;
- сохраняет их;
- возвращает их обратно через API.

Именно Stage 01 впервые связывает вместе:

- HTTP;
- доменную сущность;
- persistence;
- migrations;
- тесты.

## Что обычно делают наивно

### Наивный вариант 1: писать все прямо в controller

Например:

- читать request;
- собирать entity;
- сохранять entity через repository;
- сразу возвращать ответ.

Почему это возникает:

- так быстрее написать первый working flow;
- студенту кажется, что controller “и так все видит”.

Проблемы:

- HTTP-слой знает слишком много;
- бизнес-логика смешивается с API-логикой;
- переиспользование почти невозможно;
- тестировать сложнее;
- при росте требований controller разрастается первым.

### Наивный вариант 2: отдавать entity напрямую в API

Почему так делают:

- меньше классов;
- кажется, что DTO “только мешают”.

Проблемы:

- внешняя модель жестко сцепляется с внутренней;
- сложнее менять структуру БД без изменения API;
- возрастает риск утечки лишних полей;
- студент не видит разницы между транспортной моделью и доменной моделью.

### Наивный вариант 3: не использовать migration

Почему так делают:

- проще быстро создать таблицу вручную;
- кажется, что “на маленьком проекте можно без этого”.

Проблемы:

- нет воспроизводимости среды;
- непонятно, какая схема считается правильной;
- тесты и локальная среда начинают расходиться;
- дальнейшее развитие БД становится хаотичным.

## Почему наивный подход здесь уже недостаточен

На Stage 01 у нас уже появляется устойчивый вертикальный use case.

Это значит, что нам уже нужна минимальная инженерная дисциплина:

- controller отвечает за HTTP;
- service отвечает за поведение;
- repository отвечает за доступ к данным;
- DTO отвечает за внешнюю форму данных;
- entity отвечает за внутреннюю форму хранения;
- migration отвечает за воспроизводимую схему.

Если этого не сделать сейчас, уже на Stage 02 студент начнет путаться, где должна жить новая логика.

## Профессиональный подход на этом этапе

### 1. Разделить ответственность по слоям

[TaskDefinitionController.java](./start/src/main/java/com/taskmanager/controller/TaskDefinitionController.java)

Controller:

- принимает запрос;
- запускает validation;
- делегирует сервису;
- возвращает HTTP response.

[TaskDefinitionService.java](./start/src/main/java/com/taskmanager/service/TaskDefinitionService.java)

Service:

- координирует работу use case;
- должен содержать реализацию `create`, `getById`, `getAll`.

[TaskDefinitionRepository.java](./start/src/main/java/com/taskmanager/repository/TaskDefinitionRepository.java)

Repository:

- знает только про persistence operations;
- не должен решать, как строится API.

### 2. Разделить DTO и entity

[TaskDefinitionRequest.java](./start/src/main/java/com/taskmanager/dto/TaskDefinitionRequest.java)  
[TaskDefinitionResponse.java](./start/src/main/java/com/taskmanager/dto/TaskDefinitionResponse.java)  
[TaskDefinition.java](./start/src/main/java/com/taskmanager/entity/TaskDefinition.java)

Почему это лучше:

- можно независимо развивать API и схему хранения;
- легче объяснить границы системы;
- студент сразу видит, что “форма данных на границе” и “форма данных в БД” — это не одно и то же.

### 3. Выделить mapper

[TaskDefinitionMapper.java](./start/src/main/java/com/taskmanager/mapper/TaskDefinitionMapper.java)

Почему это оправдано уже сейчас:

- преобразование уже есть;
- оно достаточно стабильное;
- оно не должно размазываться по controller и service.

Это не “паттерн ради красоты”, а простое средство удержать код в читаемой форме.

### 4. Использовать migration как источник правды по схеме

[V1__create_task_definitions.sql](./start/src/main/resources/db/migration/V1__create_task_definitions.sql)

Это профессионально лучше ручного создания таблиц, потому что:

- среда воспроизводима;
- схема живет рядом с кодом;
- тестовая и рабочая логика держатся ближе друг к другу;
- изменения БД становятся частью поставки кода.

## Почему этот подход оправдан именно сейчас

На Stage 01 уже оправданы:

- controller;
- service;
- repository;
- DTO;
- entity;
- mapper;
- migration.

Но еще не оправданы:

- сложные policy objects;
- orchestration;
- отдельная execution model;
- pagination;
- расширяемые task types;
- поведенческие паттерны вроде Strategy.

То есть этот этап важен тем, что он учит не только “что добавить”, но и “чего не надо добавлять слишком рано”.

## Разбор плохого и хорошего решения

### Плохой вариант

Controller:

- принимает request;
- сам создает entity;
- сам дергает repository;
- сам строит response;
- сам обрабатывает not found и validation edge cases.

Следствия:

- контроллер быстро превращается в центр логики;
- невозможно ясно объяснить слои;
- следующая сущность почти наверняка будет сделана так же хаотично.

### Лучший вариант

Controller тонкий, service координирует поведение, mapper отвечает за преобразование, repository хранит данные.

Выигрыш:

- легче читать;
- легче расширять;
- легче тестировать;
- легче объяснить, куда класть новую логику.

Цена:

- больше файлов;
- требуется дисциплина;
- студенту нужно понять несколько ролей вместо одной.

Эта цена оправдана, потому что уже на следующем этапе появится вторая сущность, и без этих границ код начнет быстро расползаться.

## Как обычно пишут код плохо и к чему это приводит

### Плохо: entity напрямую в API

Следствие:

- жесткая связка между БД и внешним контрактом;
- труднее безопасно менять модель.

### Плохо: repository используется как “главный сервис”

Следствие:

- пропадает уровень координации поведения;
- бизнес-логика начинает жить не там, где ее удобно искать.

### Плохо: validation откладывают “на потом”

Следствие:

- некорректные данные начинают проникать глубже в систему;
- ошибки всплывают слишком поздно;
- API становится менее предсказуемым.

## Антипаттерны этапа

### Антипаттерн 1: “Давайте сразу полный CRUD и все возможные фичи”

Следствие:

- студент теряет фокус;
- первый важный vertical slice размывается.

### Антипаттерн 2: “DTO не нужны, проект маленький”

Следствие:

- смешиваются границы модели;
- позже труднее объяснять эволюцию API.

### Антипаттерн 3: “Миграции потом”

Следствие:

- уже ранний этап курса становится нерепродуцируемым.

## Что студент должен понять после этой лекции

Студент должен понимать:

- зачем нужны слои именно на этом этапе;
- почему DTO уже оправданы;
- почему migration — часть инженерной дисциплины, а не лишняя бюрократия;
- почему мы все еще не вводим более тяжелые архитектурные конструкции.
